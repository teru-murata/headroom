LLM providers cache prompt prefixes to avoid reprocessing identical input on
repeated calls. Headroom treats provider cacheability as a constraint: stable
prefix and protected-prefix content should be preserved unless an explicit
future feature changes that policy.

The current local path is conservative. CacheAligner detects volatile prefix
content and reports cache-stability metrics; the cache-aware net-savings model
estimates whether compression should be skipped when cache miss penalty or CCR
retrieve cost outweighs raw token reduction.

## How CacheAligner works

System prompts often contain dynamic content -- today's date, session IDs, timestamps -- that changes between requests. Even a single character difference at the start of a prompt can invalidate a provider prefix cache.

The detector identifies these risks so callers can protect stable prefixes or
make an explicit cache-aware decision. True transport-level delta mode or
automatic prefix rewriting remains future work.

```
Before:
  "You are helpful. Current Date: 2025-04-06"  <- changes daily, no cache hit

Potential policy outcome:
  "You are helpful."                            <- stable prefix, cache hit
  "[Context: Current Date: 2025-04-06]"         <- dynamic part handled outside the protected prefix
```

The example above illustrates a cache-stability goal, not a guarantee that the
current Headroom pipeline rewrites every prompt prefix.

## Provider-specific strategies

Each LLM provider implements caching differently. Headroom tracks these
differences as local configuration and benchmark inputs; it does not guarantee
provider cache hits or exact billing outcomes.

### Anthropic

Anthropic supports explicit `cache_control` blocks that mark content as cacheable. Cached input tokens cost **90% less** than regular input tokens.

Provider integrations can use explicit `cache_control` breakpoints where the
caller enables that behavior. Keep stable prefixes intact when cacheability
matters.

| Metric | Value |
|---|---|
| Cache read discount | 90% off input price |
| Cache write cost | 25% premium on first write |
| Cache TTL | 5 minutes (extended on hit) |

### OpenAI

OpenAI uses automatic **prefix caching** -- if consecutive requests share the same message prefix, the provider reuses cached KV states. No explicit API markers are needed, but the prefix must be byte-identical.

OpenAI prefix caching depends on byte-identical prefixes. Headroom's local
cache-aware policy should avoid rewriting protected prefixes that may already be
cacheable.

| Metric | Value |
|---|---|
| Cache read discount | 50% off input price |
| Activation | Automatic (prefix match) |
| Min prefix length | 1024 tokens |

### Google

Google provides the **CachedContent API**, which lets you explicitly cache large context (system instructions, documents, tools) and reference it across requests. Cached tokens cost **75% less**.

Provider-side cached-content lifecycle is provider-specific and should be
configured explicitly. Hosted pricing catalogs and ROI reporting belong to
coding-agent-bridge, not this local Headroom repository.

| Metric | Value |
|---|---|
| Cache read discount | 75% off input price |
| Mechanism | Explicit CachedContent API objects |
| Min cache size | 32,768 tokens |

## Configuration

<Tab value="TypeScript">
```ts twoslash
import { compress } from "headroom-ai";
import type {
  CacheAlignerConfig,
  CacheOptimizerConfig,
  HeadroomConfig,
} from "headroom-ai";

// CacheAligner: detect volatile prefix content and cache-risk patterns
const cacheAligner: CacheAlignerConfig = {
  enabled: true,
  datePatterns: [
    "Today is \\w+ \\d+, \\d{4}",
    "Current time: .*",
  ],
  normalizeWhitespace: true,
  collapseBlankLines: true,
};

// CacheOptimizer: provider-level caching
const cacheOptimizer: CacheOptimizerConfig = {
  enabled: true,
  autoDetectProvider: true,  // Detect Anthropic/OpenAI/Google automatically
  minCacheableTokens: 1024,
};

// Full configuration
const config: HeadroomConfig = {
  cacheAligner,
  cacheOptimizer,
};

// Compress with cache-aware local policy
const result = await compress(messages, {
  model: "claude-sonnet-4-20250514",
  config,
});
```

```python
from headroom import HeadroomClient, OpenAIProvider, AnthropicProvider, GoogleProvider
from headroom.transforms import CacheAlignerConfig
from openai import OpenAI

# CacheAligner configuration
aligner_config = CacheAlignerConfig(
    enabled=True,
    dynamic_patterns=[
        r"Today is \w+ \d+, \d{4}",
        r"Current time: .*",
        r"Session ID: [a-f0-9-]+",
    ],
)

# Provider-specific cache settings
# OpenAI: prefix caching (automatic, just keep prefixes stable)
client = HeadroomClient(
    original_client=OpenAI(),
    provider=OpenAIProvider(enable_prefix_caching=True),
    enable_cache_optimizer=True,
)

# Anthropic: cache_control blocks (90% read discount)
from anthropic import Anthropic

client = HeadroomClient(
    original_client=Anthropic(),
    provider=AnthropicProvider(enable_cache_control=True),
    enable_cache_optimizer=True,
)

# Google: CachedContent API (75% read discount)
client = HeadroomClient(
    original_client=google_client,
    provider=GoogleProvider(enable_context_caching=True),
    enable_cache_optimizer=True,
)
```

## How estimates compound

Cache-aware policy and provider caching interact with Headroom's compression transforms:

1. **Compression transforms** may reduce eligible noisy context.
2. **Cache-aware policy** should protect stable provider prefixes.
3. **Provider caching** may discount cacheable input when the provider accepts it.
4. **CCR retrieval** can add estimated follow-up cost if omitted content is later retrieved.

For local decision support, see [Cache-Aware Net Savings](/docs/cache-aware-net-savings). It models gross saved tokens, cache miss penalty, CCR retrieve cost, and net saved tokens. The result is not exact provider billing and not an invoice denominator.
